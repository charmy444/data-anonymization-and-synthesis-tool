# Архитектура репозитория

## Цель
Зафиксировать структуру проекта, ответственность модулей и правила, чтобы команда одинаково раскладывала новую логику и не дублировала код.

## Слои и назначение

### `src/sda/core/generation/`
Доменная логика генерации данных.
- Генерация строк по шаблонам таблиц.
- Правила связей между сущностями (например, `user_id` в зависимых таблицах).
- Никакой зависимости от FastAPI, HTTP, UploadFile, Response.

### `src/sda/core/anonymization/`
Доменная логика анонимизации.
- Применение правил обработки к колонкам и значениям.
- Реализация стратегий анонимизации.
- Никакой зависимости от web-слоя и Pydantic API-моделей.

### `src/sda/core/ml/`
Эвристики и синтез похожих данных.
- `suggest_pii`: эвристическая подсказка потенциально чувствительных колонок.
- `similar/synthesize`: анализ структуры/статистик и генерация похожего synthetic CSV.
- Не содержит HTTP-логики и не работает напрямую с endpoint-контрактами.

### `src/sda/io/`
Чтение и запись CSV.
- Унифицированный вход/выход CSV для приложения.
- Парсинг входных CSV и сериализация выходных CSV.
- Без бизнес-решений уровня "как анонимизировать" или "что генерировать".

### `src/sda/use_cases/`
Сценарии уровня приложения (application layer).
- Связывают `core`, `io` и web-вызов.
- Оркестрация шагов: чтение данных -> доменная обработка -> подготовка результата.
- Точка, где должна жить "сквозная" логика фичи.

### `src/sda/web/routers/`
HTTP endpoints.
- Принимают запросы, вызывают use_case, возвращают HTTP-ответ.
- Не реализуют доменную логику генерации/анонимизации/синтеза внутри роутера.
- Минимальная ответственность: маршрутизация, коды ответов, базовая обработка ошибок.

### `src/sda/web/schemas/`
Pydantic-схемы API.
- Контракты входа/выхода endpoint'ов.
- Валидация полей запроса и структур ответа.
- Не содержат бизнес-вычислений.

### `src/sda/resources/templates/`
Шаблоны генерации.
- Описания таблиц и колонок для генератора.
- Источник для `Generate`-ветки без хардкода схем в роутерах.

### `tests/`
Модульные и API-тесты.
- Unit-тесты для `core`, `io`, `use_cases`.
- API-тесты для `web/routers` и контрактов.
- Проверка, что поведение фич совпадает с scope и контрактами.

## Правила зависимостей (обязательные)

1. `core` ничего не знает о FastAPI.
2. `web` не генерирует и не анонимизирует данные сам, а только делегирует в `use_cases`.
3. `use_cases` связывают `core`, `io` и `web`.
4. `io` не решает бизнес-задачу, только преобразует CSV.
5. `schemas` используются как API-контракты, а не как замена доменных моделей.

## Куда добавлять новую логику

### Фича `suggest_pii`
- Эвристика: `src/sda/core/ml/pii_suggester.py`
- Сценарий: `src/sda/use_cases/suggest_pii.py`
- HTTP endpoint: `src/sda/web/routers/suggest.py`
- API-схемы (если нужны): `src/sda/web/schemas/`
- Тесты: unit в `tests/test_suggest.py`, API в отдельном `test_api_*`.

### Фича `similar`
- Алгоритм/синтез: `src/sda/core/ml/synthesizer.py` (или связанный модуль в `core/ml`).
- Сценарий: `src/sda/use_cases/synthesize_csv.py`
- HTTP endpoint: `src/sda/web/routers/similar.py`
- API-схемы: `src/sda/web/schemas/similar.py`
- Тесты: `tests/test_synthesis.py` и `tests/test_api_similar.py`.

## Анти-паттерны
- Дублировать одну и ту же бизнес-логику одновременно в `router` и `use_case`.
- Читать/писать CSV прямо в роутере, минуя `io`.
- Вносить HTTP-типизации и статусы ответов в `core`.
- Вшивать схемы таблиц напрямую в endpoint-код вместо `resources/templates`.
